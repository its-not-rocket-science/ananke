# Ananke — Content-Pack Registry Specification

This document describes the `registry` metadata block added to `.ananke-pack` manifests
in v0.1.64 (PM-6).  A pack that includes a `registry` block can be:

- **verified** for engine compatibility at load time,
- **integrity-checked** via its SHA-256 checksum,
- **licensed** with a standard SPDX identifier, and
- **provenance-attested** with dataset or paper references.

---

## The `registry` block

Add a `registry` object to the top level of your pack manifest:

```json
{
  "name": "weapons-historical",
  "version": "1.2.0",
  "description": "Peer-reviewed medieval weapon profiles.",
  "registry": {
    "compatRange": ">=0.1.60",
    "stabilityTier": "stable",
    "requiredExports": ["./catalog", "./combat"],
    "checksum": "<64-char SHA-256 hex — generated by `npx ananke pack bundle`>",
    "license": "CC-BY-4.0",
    "provenance": [
      {
        "title": "Thermite & Clements (2011): Sword performance metrics",
        "doi": "10.1093/jhmas/jrq001",
        "notes": "Grounds `strikeSpeedMul` and `strikeEffectiveMassFrac` for longsword class."
      }
    ]
  },
  "weapons": [ ... ]
}
```

## Field reference

### `compatRange` (string, optional)

A [semver range](https://semver.org/) string describing which Ananke engine versions this
pack targets.  `validatePack` rejects packs whose `compatRange` does not include the
running engine version.

Supported range syntax:

| Syntax | Meaning |
|--------|---------|
| `>=0.1.50` | any version ≥ 0.1.50 |
| `>0.1 <0.2` | space-separated compound — all constraints must match |
| `^0.1.60` | compatible with major: `>=0.1.60 <0.2.0` |
| `~0.1.60` | compatible with minor: `>=0.1.60 <0.1.61` — use for strict patch pins |
| `=0.1.64` | exact version only |

### `stabilityTier` (`"stable"` | `"experimental"` | `"internal"`)

Controls how the pack appears in a public registry:

| Tier | Meaning |
|------|---------|
| `stable` | Production-ready; semver compatibility guaranteed within major version |
| `experimental` | In development; breaking changes possible without notice |
| `internal` | Not intended for external use; excluded from public registry listings |

### `requiredExports` (string[], optional)

Subpath exports from `@its-not-rocket-science/ananke` that this pack's content uses,
e.g. `["./catalog", "./combat"]`.  Informational — a future registry can use this to
warn when a host's available exports don't match the pack's requirements.

### `checksum` (string, optional)

SHA-256 hex digest of the pack JSON for integrity verification.

**How it is computed:**

1. Set `registry.checksum` to `""` (the field must be present but blank).
2. Serialise the entire manifest to JSON with `JSON.stringify(manifest, null, 2)`.
3. Take the SHA-256 of that string.
4. Write the 64-character lowercase hex digest back into `registry.checksum`.

The `npx ananke pack bundle <directory>` command does this automatically.

**To verify a checksum manually:**

```typescript pseudocode
import { createHash } from "node:crypto";
import { readFileSync } from "node:fs";

const raw = JSON.parse(readFileSync("my-pack.json", "utf8"));
const stored = raw.registry?.checksum ?? "";
raw.registry.checksum = "";
const actual = createHash("sha256")
  .update(JSON.stringify(raw, null, 2))
  .digest("hex");
console.log(stored === actual ? "✓ checksum ok" : "✗ checksum mismatch");
```

### `license` (string, optional)

[SPDX license identifier](https://spdx.org/licenses/) for the pack content.
Examples: `"MIT"`, `"CC-BY-4.0"`, `"Apache-2.0"`, `"Proprietary"`.

### `provenance` (object[], optional)

Array of source references for empirically grounded content.  Each entry has:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | yes | Short description of the source |
| `url` | string | no | URL of the source |
| `doi` | DOI string | no | DOI of the paper or dataset |
| `notes` | string | no | What this source grounds in the pack |

## Runtime enforcement

`validatePack(manifest)` performs the following registry checks:

| Check | Condition | Error path |
|-------|-----------|------------|
| `registry` is object | if present | `$.registry` |
| `compatRange` satisfaction | engine version must satisfy the range | `$.registry.compatRange` |
| `stabilityTier` value | must be `stable`, `experimental`, or `internal` | `$.registry.stabilityTier` |
| `requiredExports` is array of strings | if present | `$.registry.requiredExports[i]` |
| `checksum` is 64-char hex | if present | `$.registry.checksum` |
| `license` is non-empty string | if present | `$.registry.license` |
| `provenance` is array with `title` | if present | `$.registry.provenance[i].title` |

The `checksum` field is **not** verified by `validatePack` — checksum verification is a
host responsibility, since hashing a large manifest in a browser context is expensive.
Use the CLI or the snippet above to verify integrity when loading packs from untrusted
sources.

## Future online registry

A future `registry.ananke.dev` can index packs by reading their `registry` block:

1. **Indexing**: crawler fetches the pack JSON, verifies `checksum`, reads metadata.
2. **Filtering**: host queries `?compatRange=>=0.1.60&stabilityTier=stable`.
3. **Provenance**: research-backed packs surface in a "peer-reviewed" filter.
4. **Licensing**: hosts can exclude non-commercial packs from their build pipeline.

The `registry` block is designed to be sufficient for this future without any breaking
changes to the manifest format.

## Regenerating fixtures

After any change to the registry format, regenerate the conformance fixtures:

```bash

```
